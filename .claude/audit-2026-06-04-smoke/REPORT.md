### Cluster B — episodeStorage dead code (mechanical)

**Findings:** F004, F005, F006, F007, F009

**Why it matters:** Five low-severity findings concentrated in one in-scope file. F004/F005 are two genuinely unused exports (`deleteEpisode`, `getActiveEpisodes` — the latter superseded by `getVisibleDashboardEpisodes`). F006/F007/F009 are three independent reports of the **same** dead `queueMicrotask` no-op block in `allocateEpisodeSequence` — it allocates a throwaway Promise per call, reads as if it prunes `_episodeSequenceLocks`, and prunes nothing (a maintenance trap).

**Plan:**
1. Delete the `queueMicrotask(...)` no-op block (≈ lines 148–155), keeping only `release()` in the `finally`. This resolves F006, F007, and F009 in one edit. If genuine lock-map cleanup is wanted, instead store the chain in a local and `_episodeSequenceLocks.delete(episodeId)` against that exact reference.
2. Delete `deleteEpisode` (F004, ≈ lines 189–200) — only `clearAllEpisodes` is used for removal.
3. Delete `getActiveEpisodes` (F005, ≈ lines 217–235) — `getVisibleDashboardEpisodes` is the live equivalent.

**Risk:** Low but verify-before-delete. Confirm grep returns zero importers for `deleteEpisode` / `getActiveEpisodes` before removal (the finding already did, but re-check after any concurrent edits). The `queueMicrotask` block is provably inert (`prior.then(...)` allocates a fresh Promise, so the equality is always false).

**Verification:** `npm run check:types` → `npm run test` → `npm run lint`. Ideally land **after** Cluster C so the new episodeStorage tests guard the allocateEpisodeSequence edit.

### Cluster C — episodeStorage test coverage (mechanical)

**Findings:** F002

**Why it matters:** `episodeStorage.ts` (322 lines) has zero coverage despite containing clinical-attribution-relevant logic: `allocateEpisodeSequence`'s concurrency mutex (stamps `episodeSequence` onto cases), `updateEpisode`'s `EPISODE_STATUS_TRANSITIONS` guard, and `getVisibleDashboardEpisodes`'s 7-day linger filter. All three are deterministic given mocked storage. Adding this suite also de-risks the Cluster B deletion of the dead block inside `allocateEpisodeSequence`.

**Plan:** Add `client/lib/__tests__/episodeStorage.test.ts`. Mock `@react-native-async-storage/async-storage` (in-memory Map), `./encryption` (identity passthrough), `./storage` (hashPatientIdentifier, getCasesByEpisodeId), `./activeUser` (userScopedAsyncKey) — the same pattern existing storage tests use. Cover:
- `allocateEpisodeSequence` returns 1 for a missing episode, increments across sequential calls, and serialises two concurrent un-awaited calls into distinct sequence numbers (the mutex invariant).
- `updateEpisode` rejects an illegal transition (e.g. completed→planned) while still persisting a co-submitted non-status field; applies a legal transition.
- `getVisibleDashboardEpisodes` includes active/on_hold/planned, excludes cancelled, includes a completed episode resolved within 7 days, excludes one resolved >7 days ago or with no resolvedDate.

**Risk:** Very low — additive test file, no production code change.

**Verification:** `npm run test -- episodeStorage` then full `npm run test:coverage`. Confirm the new file lifts episodeStorage coverage off zero.

### Cluster D — drizzle-orm major upgrade (logic)

**Findings:** F001

**Why it matters:** The only non-mechanical finding. `drizzle-orm@0.39.3` carries an unpatched HIGH SQL-injection advisory in the deployed server runtime (`server/storage.ts` uses Drizzle for every DB query). The fix is a semver-major bump to `0.45.2` that `npm audit fix` won't apply without `--force`. Needs a deliberate, tested upgrade — not a blind run.

**Plan:**
1. Audit `server/storage.ts` for any dynamic/attacker-influenced column or table identifier interpolation (Opus is believed to use static schema identifiers — confirm this to validate the low-exploitability assumption).
2. Bump `drizzle-orm` 0.39 → 0.45, run `drizzle-kit push` against a **scratch** DB, and review the [drizzle 0.40–0.45 migration notes](https://orm.drizzle.team) for breaking changes to query/schema APIs.
3. Run the server `__tests__` suite + client typecheck.

**Risk:** Medium — semver-major ORM bump touches every query path. Mitigate with scratch-DB push and the full server test suite before deploying to Railway. **Not** an auto-apply candidate for the mechanical remediation wave.

**Verification:** `npm run check:types` → server `npm run test` → `drizzle-kit push` against scratch DB → manual smoke of one read + one write endpoint locally before `railway up`. Re-run `npm audit --omit=dev` to confirm the HIGH advisory clears.
